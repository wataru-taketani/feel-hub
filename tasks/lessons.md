# 教訓・再発防止ログ

## 2026-03-09: user_profiles カラム名 id vs user_id 事故

### 事象
FC同期アーキテクチャ導入で、5ファイルで `user_profiles` テーブルに `.eq('user_id', ...)` と書いたが、
実際のPKカラム名は `id`。PostgRESTはエラーを返さず空結果を返すため、サイレントに失敗。
ダッシュボードの予約が全て非表示になった。

### 根本原因
- テーブルごとのカラム名の違いを確認せず、他テーブルの `user_id` パターンを流用した
- `npm run build` はDBスキーマ整合性を検証しない
- デプロイ後の本番スモークテスト（実際に画面を開く）を省略した

### 再発防止ルール
1. **DB操作コードを書く前に、対象テーブルのカラム名を必ず確認する**
   - `SELECT column_name FROM information_schema.columns WHERE table_name = 'xxx'` で確認
   - 特に `user_profiles` は PK が `id`（`user_id` ではない）
2. **新規APIや既存API大幅変更時、デプロイ後に必ずDB応答を確認する**
   - Supabase execute_sql で `SELECT count(*) FROM <table> WHERE <condition>` を実行
   - レスポンスが空でないことを検証
3. **複数テーブルを扱う変更では、テーブルごとのカラム命名規則を一覧化してからコーディングする**

### テーブル別カラム名一覧（user識別子）
| テーブル | user識別カラム | 備考 |
|---|---|---|
| user_profiles | `id` | PK = auth.users.id |
| feelcycle_credentials | `user_id` | FK |
| user_reservations | `user_id` | FK |
| user_tickets | `user_id` | FK |
| attendance_history | `user_id` | FK |

## 構造的教訓（2026-03-09 総括）

以下は個別のバグではなく、開発プロセスの構造的欠陥から学んだ教訓。
CLAUDE.md の開発ルールに反映済み。形骸化していないか定期的に振り返ること。

### 教訓1: ビルド成功 ≠ 品質保証
- `npm run build` はTypeScript型エラーとNext.jsビルドエラーしか検出しない
- DBカラム名の不一致、PostgRESTの空結果、APIレスポンスの内容はビルドで検証不可能
- **対策**: デプロイ後に必ず本番DBクエリでデータ存在を確認する。ビルド成功だけで「完了」と報告しない

### 教訓2: 変更規模とリスクは比例する
- 11ファイルを同時変更（FC同期アーキテクチャ導入）で、同じ間違いが5ファイルに伝播した
- 1ファイルの変更と11ファイルの変更では、必要な検証レベルが全く異なる
- **対策**: 変更規模をS/M/Lで分類し、M以上はコーディング前チェックとPlan modeを必須にする

### 教訓3: デプロイ報告にはエビデンスが必要
- 「修正しました」「デプロイしました」だけでは、実際に動作しているか不明
- PostgRESTはカラム名不一致でもエラーを返さないため、本番で初めて問題が顕在化する
- **対策**: 完了報告にはSQLクエリ結果やログ出力など、検証のエビデンスを必ず含める

### 教訓4: 本番DBのデータは追跡可能にする
- テストデータを識別不能な形で本番DBに投入し、後から区別できなくなった
- 同一セッション内でクリーンアップしなかったため、不要データが残存した
- **対策**: 本番DBへのテストデータ投入は原則禁止。やむを得ない場合は識別子を付与し、同一セッション内で削除する

### 教訓5: 障害対応は「全体調査 → 一括修正」
- 1箇所の修正で終わらせると、同じ原因の問題が他に残る（5ファイル中1ファイルだけ直すなど）
- 五月雨式に1件ずつ対応すると、漏れが生じやすく、ユーザーの信頼を損なう
- **対策**: 障害発生時はまず全体の影響範囲を特定し、同じパターンを全件検索してから一括修正する

### 教訓6: 異なるデータソースの値を「同じだろう」と仮定しない
- **事象（2026-03-15）**: `reservedMap` のキーに `studio` を追加したら予約済み表示が全て消えた。`lessons.studio = "川崎"` vs `user_reservations.studio = "川崎（KWS）"` でフォーマットが異なっていた
- **根本原因**:
  1. 2つのテーブルの `studio` 値を `SELECT DISTINCT` で確認せずに比較キーに追加した
  2. 「同じカラム名だから同じフォーマットだろう」という仮定で実装した
  3. S規模と判定してコーディング前チェックを省略した（実際はデータソース間の比較導入でM規模）
  4. ビルド成功だけで検証完了とした
- **対策**:
  1. 異なるテーブルの値を比較する変更は最低M規模。事前に `SELECT DISTINCT` で実データ確認必須
  2. 文字列フィールドの比較より UUID（lesson_id 等）による結合を優先する
  3. 「1フィールド追加するだけ」でも、データパイプラインが異なれば高リスク変更

### 教訓7: 「たぶんこれ」でデプロイしない — 要件理解の浅さがリソース浪費を招く
- **事象（2026-03-10）**: スタジオフィルタ変更で3回pushが発生。1回目でStudioSelectDialogを閉じられなくする実装 → ユーザー「選ばせるな、ホームストアにしろ」→ 2回目で修正。CLAUDE.md更新も2回に分割。結果Vercel無料枠（4h CPU/月）を使い切った
- **根本原因**:
  1. ユーザーの意図（「初期設定は所属店舗」）を「ダイアログで強制選択」と誤解した
  2. チームレビューがコード正確性のみで、要件一致・UX妥当性を検証しなかった
  3. 「確信がないまま、たぶんこれで合っている」でデプロイした
- **対策**:
  1. 実装前にユーザーの要件を自分の言葉で言い換え、解釈にズレがないか確認する
  2. チームレビューに「要件一致」「UX妥当性」の観点を必須化する
  3. 1タスク1デプロイの原則を徹底。ドキュメント変更もコード変更と同一コミットにまとめる
  4. 不確実なら実装前にユーザーに確認する。推測でデプロイしない

## 2026-03-12: proxy.ts死んだコード事故 + Vercel使用量問題

### 事象
- Vercel Hobbyプランの全リソース（CPU 4h, Invocations 1M, Edge Requests, FOT）が上限に到達
- `proxy.ts` がNext.jsに認識されず、middlewareが数ヶ月間動いていなかった
- ボット/クローラーがTOPページにアクセス→JS実行→useEffectで3API並列呼び出し→1訪問=4リクエスト消費
- `/api/mypage` の `totalAttendance` がハードコード0だった
- `/api/history/sync` にFC自動再認証が未導入だった

### 根本原因
- Next.jsは `middleware.ts` のみ認識するが、`proxy.ts` という名前で放置されていた
- robots.txtがなく、ボットのアクセスを抑制できていなかった
- 全ページログイン不要だったため、ボットがページのJSを実行しAPI呼び出しを発生させた
- 新API追加時に既存パターン（getFcSession/reauthSession）の適用漏れ

### 対策
1. `proxy.ts` → `middleware.ts` にリネーム
2. 全ページログイン必須化（ボットはログイン画面で停止）
3. matcherを認証必須ルートのみに限定（Edge Requests削減）
4. Cookieなしリクエストは `getUser()` スキップ（Supabase通信回避）
5. Supabase APIエラー時はログアウトさせない安全策
6. robots.txt追加
7. 履歴同期APIにFC自動再認証導入
8. マイページ累計受講回数をDBから取得

### 再発防止ルール
1. **Next.jsの規約ファイル名は正確に**: `middleware.ts`, `layout.tsx`, `page.tsx` 等は命名が重要。リネームや移動時は動作確認必須
2. **新API追加時は既存パターンを確認**: FC自動再認証（getFcSession/reauthSession）のように、プロジェクト内の確立されたパターンを踏襲する
3. **ハードコード値は疑う**: API応答に0やnullをハードコードしている箇所は、本来DBから取得すべきか確認する
